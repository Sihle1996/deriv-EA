## What this project is

An automated trading bot for **Deriv synthetic indices**, to be driven (eventually) by a
**React dashboard** via the **Deriv WebSocket API**. Built solo, starting strictly on a
**DEMO** account. The user has an Angular / Spring / .NET background and is comfortable with React.

## ⚠️ The honest caveat (do not lose this)

Deriv synthetic indices are generated by a **cryptographically secure RNG**. There is **no order
flow, news, or liquidity** to read, so chart patterns (incl. the "Forex Master Pattern" the user
is interested in) **do not carry a predictive edge** here. Multi-timeframe views are statistically
self-similar noise — they organise logic and trade frequency but do **not** stack independent
edges. Build this as honest infrastructure + a research harness, **not** a profit guarantee.
Always be straight with the user about this; never imply a bot will reliably beat the house edge.

## Decisions locked

- **Phased build.** Data → Signal → Log → Review → Trade. Never jump straight to trading.
- **Phase 1 runs UNAUTHENTICATED** (`DERIV_AUTHENTICATE=false`, default). Legacy v3 market data
  (ticks/candles/`active_symbols`) is **public** — no token needed. When `authenticate=true`
  (later phases), `main.py` HARD-BLOCKS real-money accounts (raises if `is_virtual` false).
- **Connection:** raw `websockets` (transparent, minimal deps). The official SDK can slot in later
  for the trade loop if reconnection logic gets fiddly.
- **Symbol:** Step Index (`stpRNG` default; `main.py` lists all step symbols on startup to confirm).
- **Multi-timeframe:** ONE tick stream + ONE base (1m) candle stream; higher TFs (5m/15m) are
  **resampled in-process** with pandas (fewer subscriptions, perfectly aligned bars).
- **Persist every raw tick** to daily Parquet — ticks are the archive of record (candles are
  derivable from ticks, not vice-versa) so any future pattern definition can be re-tested.
- **`MarketSnapshot`** (in `candles.py`) is the ONLY surface downstream/strategy code may consume.
  Phase 2 must never touch pandas / websocket messages directly.

**Bug fixed during bring-up:** `deriv_client.run()` originally awaited `on_connect` *then* the read
loop — a deadlock, since `authorize`/`subscribe` await reply futures only the read loop resolves.
Fixed by running the read loop as a concurrent task started *before* `on_connect`. Don't reintroduce.

**Gap-free archive (added after first soak found ~40s reconnect gaps):**
- Faster drop detection: `ping_interval`/`ping_timeout` = 10s (was 20s) → smaller pre-reconnect gaps.
- **Reconnect backfill:** on every (re)connect, `Spine._backfill_ticks` refetches the missed window
  via `ticks_history` and splices it in. It LOOPS to defeat the history-endpoint lag (Deriv's
  `ticks_history` 'latest' trails the live stream by several seconds), backfilling up to the first
  live tick so history meets live with no seam. Verified: backfill reached epoch N, live from N+1.
- **Restart backfill:** `seed_last_epoch_from_archive()` seeds `last_epoch` from the newest archived
  tick on startup (if within the backfill window) so a kill+restart is backfilled too.
- Audit anytime with `python check_archive.py` (median spacing, gaps >5s, coverage %).

**Known limits:** ungraceful kill still loses up to `tick_flush_every-1` (≈99) *buffered* ticks, but
the next start backfills them (within the ~83-min/5000-tick `backfill_count` cap). Gaps longer than
that cap are only partially filled (logged with a WARN). The 24h soak base frame (~1440 1m candles)
never reaches `max_base_rows`=5000, so the trim path isn't exercised until ~83h.

## Auth reality (IMPORTANT — Deriv has TWO API systems)

- **Legacy WS v3** (`wss://ws.derivws.com/websockets/v3`, `{"authorize": token}`) — what this bot
  uses. Market data is **public** (no token). Needs a **classic** token for account calls.
- **New Options API** (`api.derivws.com/trading/v1/...`) — REST OTP → authenticated WS URL.
- **New Deriv accounts (incl. this user's) only mint `pat_`-prefixed tokens** for the new system.
  They return **`InvalidToken`** on the legacy `authorize` call — confirmed empirically.
- ∴ Phase 1 sidesteps auth entirely (public data). **Phase 3 trading** must implement the new OTP
  flow: `POST https://api.derivws.com/trading/v1/options/accounts/{id}/otp` with header
  `Authorization: Bearer pat_…` + `Deriv-App-ID` → response `data.url` is a ready WS URL.
- User's demo accounts: **CFDs (MT5 Standard)** and **Options**, both $10,000 virtual.

## How to run

```bash
.venv\Scripts\activate
python main.py              # NO token needed; stream + persist; Ctrl-C flushes ticks
python verify_feed.py       # feed-quality gate (also tokenless)
```

**Multi-asset background run (Phase 2 collection):** `python run_all.py` launches one DETACHED
`main.py` per symbol (edit `SYMBOLS`; currently `stpRNG` + `1HZ50V` = Step Index + Volatility 50
(1s)), logging to `logs/<symbol>.log`, PIDs in `logs/pids.txt`; `python stop_all.py` stops them
(`taskkill /T /F`, but each restart backfills the gap). Each symbol writes to its own
`data/{ticks,signals}/<symbol>/`. Launcher uses the venv **`pythonw.exe`** (windowless) on purpose:
the venv `python.exe` is a stub that spawns a worker child which POPS A CONSOLE WINDOW per bot;
`pythonw.exe` runs as a single windowless process. (Also why it's a Python launcher, not PowerShell:
`Start-Process` + output redirection throws "Item has already been added: COMSPEC" on envs with
case-variant dup env keys; Python's case-insensitive os.environ avoids it.)
