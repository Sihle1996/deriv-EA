## What this project is

An automated trading bot for **Deriv synthetic indices**, to be driven (eventually) by a
**React dashboard** via the **Deriv WebSocket API**. Built solo, starting strictly on a
**DEMO** account. The user has an Angular / Spring / .NET background and is comfortable with React.

## вљ пёЏ The honest caveat (do not lose this)

Deriv synthetic indices are generated by a **cryptographically secure RNG**. There is **no order
flow, news, or liquidity** to read, so chart patterns (incl. the "Forex Master Pattern" the user
is interested in) **do not carry a predictive edge** here. Multi-timeframe views are statistically
self-similar noise вЂ” they organise logic and trade frequency but do **not** stack independent
edges. Build this as honest infrastructure + a research harness, **not** a profit guarantee.
Always be straight with the user about this; never imply a bot will reliably beat the house edge.

## Decisions locked

- **Phased build.** Data в†’ Signal в†’ Log в†’ Review в†’ Trade. Never jump straight to trading.
- **Phase 1 runs UNAUTHENTICATED** (`DERIV_AUTHENTICATE=false`, default). Legacy v3 market data
  (ticks/candles/`active_symbols`) is **public** вЂ” no token needed. When `authenticate=true`

- **Connection:** raw `websockets` (transparent, minimal deps). The official SDK can slot in later
  for the trade loop if reconnection logic gets fiddly.
- **Symbol:** Step Index (`stpRNG` default; `main.py` lists all step symbols on startup to confirm).
- **Multi-timeframe:** ONE tick stream + ONE base (1m) candle stream; higher TFs (5m/15m) are
  **resampled in-process** with pandas (fewer subscriptions, perfectly aligned bars).
- **Persist every raw tick** to daily Parquet вЂ” ticks are the archive of record (candles are
  derivable from ticks, not vice-versa) so any future pattern definition can be re-tested.
- **`MarketSnapshot`** (in `candles.py`) is the ONLY surface downstream/strategy code may consume.
  Phase 2 must never touch pandas / websocket messages directly.

- **Reconnect backfill:** on every (re)connect, `Spine._backfill_ticks` refetches the missed window
  via `ticks_history` and splices it in. It LOOPS to defeat the history-endpoint lag (Deriv's

never reaches `max_base_rows`=5000, so the trim path isn't exercised until ~83h.

**Multi-asset background run (ATS collection):** `python run_all.py` launches one DETACHED
`main.py` per symbol (edit `SYMBOLS`; currently `stpRNG` + `1HZ50V` synthetics + `frxUSDJPY`/`frxXAUUSD`/
`OTC_NDX` real markets), logging to `logs/<symbol>.log`, PIDs in `logs/pids.txt`; `python stop_all.py`
stops them (`taskkill /T /F`, but each restart backfills the gap). Each symbol writes to its own
`data/{ticks,signals_ats}/<symbol>/`. Launcher uses the venv **`pythonw.exe`** (windowless) on purpose:
the venv `python.exe` is a stub that spawns a worker child which POPS A CONSOLE WINDOW per bot;
`pythonw.exe` runs as a single windowless process. (Also why it's a Python launcher, not PowerShell:
`Start-Process` + output redirection throws "Item has already been added: COMSPEC" on envs with
case-variant dup env keys; Python's case-insensitive os.environ avoids it.)

## ATS Master Pattern = the methodology (sole detector, NO trading)

The user trades the **TradeATS "Master Pattern"** by hand. It is now the ONLY methodology (the old
breakout detector was removed). Plan: `.claude/plans/indexed-honking-mango.md`.

The method (CANONICAL, code-audited from LuxAlgo's open-source Pine вЂ” deep-research 2026-06-16, see
below): **contraction** = a **swing-pivot compression** вЂ” a confirmed pivot-high LOWER than the prior
pivot-high AND a pivot-low HIGHER than the prior pivot-low, over `ats_pivot_lookback` bars each side
(`ta.pivothigh/ta.pivotlow`). NOT inside bars (`ats_v1` used inside bars and was wrong вЂ” too many,
~100/archive; pivot-based `ats_v2` is selective, ~39). Freeze a **value line** at the box midpoint
`(top+btm)/2` (confirmed correct), projected forward; **expansion** = a **plain** close break of a box
boundary (audited: NO ATR buffer; `ats_breakout_buffer_atr=0`); **entry** = on the LTF (1m), gated by
the **HTF bias** (HTF close vs its own value line). Gated candidates logged as `entry_blocked`.
**HTF default = 5m** (was 15m): 15m so rarely forms a pivot contraction that entries never accumulate
(1HZ50V: 26h в†’ 0); 5m defines bias far more often в†’ testable throughput. `ATS_HTF=15m` restores 15m.

**Two entry modes** (`ats_entry_mode`, both UNVERIFIED вЂ” no public ATS entry spec):
- `continuation` (default) вЂ” enter the WITH-HTF breakout's pullback to value, in the breakout direction.
- `value_fade` вЂ” the Google/TradeATS rule: enter the COUNTER-HTF expansion *spike* immediately in the
  HTF-bias direction (buy the discount / sell the premium, betting on a snap-back to value).
A/B on stpRNG: `value_fade` ~2.5Г— more entries (structural вЂ” no pullback-timeout filter), but at n=3вЂ“10
win rates are noise; only `validate_signals.py` on real-market n can compare them.

**Research caveat (important):** the audited indicators DRAW phases/boxes/lines only вЂ” **no public source
specifies entry triggers, stop-loss, take-profit, R:R, or timeframe pairing** (those are TradeATS paid
-course/marketing, unverifiable). So our HTF-gated pullback entry is OUR OWN honest heuristic, not "the"
ATS rule. And on a CSPRNG synthetic none of it carries edge by construction вЂ” faithfulness is orthogonal
to profitability.
