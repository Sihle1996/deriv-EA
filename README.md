# deriv-bot — Phase 1 (data spine) + Phase 2 (signal research)

A demo-only Deriv synthetic-indices research harness. It connects to the Deriv WebSocket API,
streams **ticks + one base candle series**, resamples higher timeframes in-process, persists
**every tick** to Parquet (gap-free across reconnects), and — Phase 2 — runs a **volatility
contraction/expansion detector** on 1m + 5m that **logs candidate signals** for offline review.
**No trading, no dashboard** — those are later phases.

> ⚠️ **Reality check.** Deriv synthetic indices are generated by a cryptographically secure RNG.
> There is no order flow, news, or liquidity to read, so chart patterns do not carry a predictive
> edge here. This project is built as honest infrastructure and a research harness — not a
> money-printing machine. Phase 2's whole point is to **measure** whether signals have any forward
> edge (via a null model); the expected, correct answer on an RNG is *no edge*. Start (and stay,
> for now) on a **demo** account.

## Requirements

- Python 3.11+ (developed on 3.13)
- **No API token needed for Phase 1.** On the legacy Deriv WebSocket API, market data
  (ticks / candles / `active_symbols`) is **public** — `authorize` is only required for account
  calls (balance, buy, portfolio), which arrive in later phases.

## Setup

```bash
python -m venv .venv
.venv\Scripts\activate        # Windows PowerShell:  .venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

That's it — `python main.py` runs immediately against public synthetic-index data.

### Authentication (later phases only)

Phase 1 runs **unauthenticated** (`DERIV_AUTHENTICATE=false`, the default). When we add
account/trading features you'll set `DERIV_AUTHENTICATE=true` and provide a token.

> **Note on Deriv's two API systems.** Newer Deriv accounts only mint **`pat_`-prefixed
> Personal Access Tokens**, which belong to Deriv's *new* Options API (a REST OTP → WebSocket-URL
> flow on `api.derivws.com`). Those tokens are **rejected** by this legacy v3 `{"authorize": …}`
> call. The trading phase will implement the OTP flow for the `pat_` token; the legacy data
> feed used here needs no token at all. When authentication is enabled, the app
> **hard-blocks real-money accounts** (exits unless `is_virtual` is true).

## Run

```bash
python main.py           # stream + persist + LOG signals; Ctrl-C to stop (flushes everything)
python verify_feed.py    # Phase 1.5 feed-quality checks (no token needed)
python check_archive.py  # audit the persisted tick archive for gaps/continuity
python verify_signals.py # Phase 2 detector regression (offline, deterministic)
python review_signals.py # Phase 2: forward outcomes + null-model comparison (the real deliverable)
python backfill_signals.py --dry-run  # regenerate the complete signal set from the tick archive
python backtest_signals.py            # simulate Rise/Fall trades + P&L vs a null model (the money question)
```

`main.py` runs both phases at once: it persists every tick **and**, on each candle close, feeds the
snapshot to the signal detector and logs any signal to `data/signals/<symbol>/<date>.jsonl`. Lines
look like `SIGNAL 1m expansion up^  bw_pct=0.18 z=-0.40 price=7998.10 bars=3`.

On reconnect or restart, `main.py` automatically **backfills** ticks missed during the outage (via
`ticks_history`), so the archive stays gap-free for backtesting — within a ~83-minute window per
outage. On startup it also prints every `step`-matching symbol from `active_symbols` so you can
confirm the exact Step Index code rather than trusting the `stpRNG` default.

### The signal logic (Phase 2)

Per timeframe (1m, 5m), a state machine **NO_SIGNAL → CONTRACTION → EXPANSION**:
- **Contraction**: Bollinger band-width drops into the low tail of its recent distribution
  (`bw_percentile <= contraction_pct`).
- **Expansion**: a later close breaks the frozen contraction range by `breakout_atr_mult × ATR`.
- Hysteresis + a contraction timeout prevent repeated firing; a warm-up gate suppresses signals
  until enough closed bars exist. All params are env-overridable (`SIG_*`, see `config.py`).

`review_signals.py` joins each logged signal against the archived ticks (look-ahead firewall:
only ticks with `epoch > bar_close_epoch`), computes forward return / MFE / MAE / first-touch, and
compares to a **null model** of random entries. If real ≈ random, there's no edge.

## Layout

| File | Role |
|------|------|
| `config.py` | env-driven config (symbol, timeframes, paths, reconnect, signal params) |
| `deriv_client.py` | raw-websockets client + auto-reconnect/resubscribe + tick backfill |
| `candles.py` | `MultiTimeframeStore` + `MarketSnapshot` (the detector surface); `TFView`/`closed_frame`/`_compute_view` (all pandas/indicator math lives here) |
| `signals.py` | pure detector — `Phase`, `SignalRecord`, `TimeframeDetector`, `SignalEngine` (no pandas/IO) |
| `storage.py` | `TickStore` → daily Parquet; `SignalStore` → daily JSONL (`data/signals/<symbol>/<date>.jsonl`) |
| `main.py` | wire-up: seed-from-archive → connect → subscribe → persist → snapshot-on-close → detect+log signals → reconnect/restart backfill |
| `verify_feed.py` | Phase 1.5: alignment / gap / resample-fidelity / tick-archive checks |
| `check_archive.py` | audits the persisted tick archive for gaps/continuity |
| `verify_signals.py` | Phase 2 detector regression (state machine, warm-up, dedup, closed-bar) |
| `review_signals.py` | Phase 2 outcome measurement: forward returns vs a null model |
| `backfill_signals.py` | regenerate the complete signal set from the gap-free tick archive (recovers any signals missed during outages; deduped vs the live log) |
| `backtest_signals.py` | contract-economics backtester — simulates Rise/Fall trades from signals, applies the payout haircut, reports P&L/ROI vs a null model (answers "would it make money?") |

## Dashboard (read-only research viewer)

A React dashboard (live candles + signal markers, recent-signal log, backtest P&L verdict, archive
health) backed by a FastAPI server. **Read-only — no trading.** The backend opens its OWN public
Deriv connection for real-time charts (independent of the collector bots) and reads
signals/backtest/health from the files the bots write.

```bash
# one-time: backend deps + frontend deps
.venv\Scripts\python -m pip install -r dashboard\requirements.txt
cd dashboard\web && npm install && cd ..\..

# dev (two terminals): backend, then the Vite dev server (proxies /api + /ws to :8000)
.venv\Scripts\python -m uvicorn dashboard.server:app --port 8000
cd dashboard\web && npm run dev      # open the printed http://localhost:5173

# OR production (one process): build the SPA, then uvicorn serves it at /
cd dashboard\web && npm run build && cd ..\..
.venv\Scripts\python -m uvicorn dashboard.server:app --port 8000   # open http://localhost:8000
```

Layout: `dashboard/server.py` (FastAPI REST + WS), `dashboard/live.py` (per-symbol live feed,
reuses `deriv_client`/`candles`), `dashboard/readers.py` (signals/backtest/health from files),
`dashboard/web/` (Vite React+TS + lightweight-charts). The dashboard discovers symbols from
`data/ticks/*`, so it shows whatever the bots are collecting.

## Phase gates

- **Phase 1 (data spine) — PASSED.** 24h soak: 100% gap-free archive, `verify_feed.py` 4/4.
- **Phase 2 (signal research) — current.** Collect signals with `main.py`, review with
  `review_signals.py`. Still **no trading**.
- **Gate before Phase 3 (trading):** > 500 reviewed signals **and** a null-model comparison showing
  a demonstrated, repeatable edge. On a CSPRNG synthetic the expected result is *no edge* → stay in
  research. That negative result is the point — it proves the patterns are noise before any money
  is risked. No trade-execution code is written until this gate is cleared.
